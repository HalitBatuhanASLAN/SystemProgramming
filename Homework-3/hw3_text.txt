Multi-Process Word Transportation and Concurrent Sorting System 
1. Overview 
In this homework, you will design and implement a multi-process system that reads 
words from a text file, distributes them across floors, transports their characters 
between floors, and reconstructs them using concurrent sorting processes. 
The goal of this assignment is not simply moving data from one place to another. Instead, 
the main objective is: 
How can multiple independent processes operate concurrently on shared data 
structures while maintaining correctness, consistency, and synchronization? 
This system includes: 
• multiple processes working simultaneously,  
• multiple active words at the same time,  
• multiple character transfers occurring concurrently,  
• multiple sorting processes operating on the same floor,  
• shared data structures accessed concurrently,  
• floor capacity constraints,  
• two independent elevator subsystems,  
• and a coordinated system-wide termination mechanism. 

---PAGE BREAK---
 
This assignment directly involves: 
• process creation  
• inter-process communication  
• shared memory  
• synchronization  
• race conditions  
• starvation, waiting, and deadlock risks  
• controlled system termination  
The design is also intentionally suitable for a thread-based extension in future work. 
2. System Scenario 
Imagine a building with multiple floors. 
You are given a .txt file containing multiple words. Each word represents a “product” 
that must be processed by the system. 
Each word: 
• has a unique ID  
• has a designated sorting floor  
• is initially placed on an arrival floor  
• and must eventually be reconstructed on its sorting floor  
Words are not transferred as a whole. Instead: 
• each word is decomposed into characters  
• characters are transported one by one  
• characters arrive in random order  
• sorting processes reconstruct the word based on its original structure  
Example: 
apple 
Characters may arrive as: 
p, e, a, l, p

Sorting processes must rearrange them correctly.

4. Process Types in the System 
4.1 Parent / Coordinator Process 
Responsible for: 
• reading the input file  
• initializing shared memory  
• creating synchronization primitives  
• spawning child processes  
• monitoring system completion  
• terminating all processes  
• printing final statistics  
• All child processes in the system must be created exclusively by the parent 
process. No other process is allowed to create system-wide processes 
• The parent process is responsible for generating the final output text file after the 
system completes 
4.2 Word-Carrier Processes 
Responsible for admitting words into the system. 
Behaviour: 
• Once created word-carrier processes never change its floor and it can bring word 
from file to only its floor where it is created. This is called arrival floor. 
• scan the input list using round-robin  
• select an unclaimed word  
• check capacity of:  
o arrival floor  
o sorting floor (from input)  
• if both are available → activate the word and put it to the arrival floor to start 
transportation operation 
• otherwise → release the word for future attempts  
This is an all-or-nothing admission control. 
 

---PAGE BREAK---
 
4.3 Letter-Carrier Processes 
Responsible for transporting characters. 
Behaviour: 
• select an available, unclaimed character task from the current floor  
• if destination differs → use delivery elevator  
• if same floor → place directly  
• after delivery → stay on destination floor  
• search for new work locally  
• if idle → use reposition elevator 
• a letter-carrier process can move away from its initial floor during execution. 
These processes are dynamic and mobile. 
If there is no letter-carrier process on any floor at any time, the parent process should 
immediately create the number of new letter-carrier processes, initially specified as an 
argument on the terminal for letter-carrier processes, on that floor. 
A letter-carrier must never process the same character task more than once, and no two 
carriers may claim the same task. 
Remember that the two things below should be done concurrently in arrival floors; 
• word-carrier processes should bring the words to the arrival floors, 
• letter-carrier processes should pick a word randomly from the arrival floors, then 
randomly get a letter of that word and start to transport to the sorting floors. 
4.4 Sorting Processes 
Operate on sorting floors. 
• multiple sorting processes per floor are allowed  
• but:  
Any sorting process in a floor can operate on any word in that floor. 
Only ONE sorting process may operate on a specific word at a time. 
 

---PAGE BREAK---
 
 
Responsibilities: 
• scan sorting area  
• fix correctly placed characters  
• move or swap incorrect ones  
• mark completion  
• never change its initial sorting floor 
 
Remember that the two things below should be done concurrently in sorting floors; 
 
• Characters of words should be randomly transported to the related area allocated 
for the word that they belong to in sorting floors and placed in the first empty place 
where letter-carrier processes find, 
• sorting processes should sort those characters based on the original word, even 
though not all letters of any word have been transported yet. 
4.5 Delivery Elevator Process 
Handles character transportation. 
Rules: 
• direction-based movement (UP / DOWN)  
• serves requests in current direction first  
• compulsory reverses at top/bottom floors (it has to changes its direction UP-
>DOWN or vice versa)  
• the elevator continues to work in the current direction until these two conditions 
are satisfied; 
o there is no received request from the floors that are located above or below 
the current floor, based on its current direction 
o the elevator contains no characters to transport(idle) 
• idle → adopts direction of first request 
Remember that the delivery elevator has a capacity indicating the maximum number of 
characters transported at any time. 

---PAGE BREAK---
4.6 Reposition Elevator Process 
Handles idle letter-carrier relocation. 
• separate from delivery elevator but same working policy as in the delivery elevator 
• used only when carriers are idle  
• moves carriers to random floors 
Remember that the reposition elevator has a capacity indicating the maximum number 
of letter-carrier processes transported at any time. 
4.7 Process Initialization 
At system startup, the parent process must create multiple processes per floor. 
• multiple word-carrier processes per floor  
• multiple letter-carrier processes per floor  
• multiple sorting processes per floor  
• one delivery elevator  
• one reposition elevator 
 
Each letter-carrier process starts execution on an assigned initial floor but may later 
move to other floors during execution. 
5. Round-Robin Word Selection 
Word-carrier processes must scan the input list in round-robin order. 
This ensures: 
• fairness  
• reduced starvation  
• predictable behavior  
Important: 
• scanning must be synchronized  
• word claiming must be atomic  

---PAGE BREAK---
6. Word Admission Policy 
A word is admitted ONLY if: 
• arrival floor has capacity  
• sorting floor has capacity  
If either fails: 
• the word is released  
• may be selected again later  
The capacity checks for arrival floor and sorting floor must be performed atomically to 
prevent inconsistent admission states. 
Pay attention that on the same floor, these two types of words can be located at the same 
time; 
• words taken from the input text file and should be transported to the different 
floors, 
• words whose arrival floor is different, but they should be sorted on that floor. 
7. Floors 
Each word has: 
Arrival Floor 
• Depending on which floor’s word-carrier process claim the word 
• initial placement  
Sorting Floor 
• defined in input  
• final reconstruction location

8. Floor Capacity 
Each floor has limited capacity: 
• maximum number of active words 
• contains currently being sorted and transported words 
If full: 
• new words cannot be admitted  
• word taken by txt should be released for future attempts 
9. Character Transportation 
Each character is a task: 
• must be claimed exactly once  
• must be delivered exactly once  
Characters are placed into sorting area: 
• NOT directly to final position  
• first empty, non-fixed slot

10. Sorting Data Structure 
Each word has: 
• original word  
• sorting_area[]  
• occupied[]  
• fixed[] 
• claimed flag 
• admitted flag  
• completed flag 
 
Set all flags to 0 initially.  
 
If a word-carrier process picks a word to bring to the arrival floor, it sets the claimed flag 
to 1.   
 
If a word-carrier process finds an available place in the arrival floor for the word that it 
brings, then the word-carrier process sets the admitted flag to 1. Otherwise, it leaves 
the admitted flag as 0 and sets the claimed flag to 0 again.  
 
If all indexes of the fixed are set to 1, then the sorting process sets the completed flag 
to 1. 
 
11. Sorting Rules 
For each slot: 
Case 1: Empty 
• do nothing  
Case 2: Fixed 
• do nothing  
Case 3: Correct position 
• mark as fixed  

---PAGE BREAK---
Case 4: Incorrect position 
• find correct target  
If: 
• target empty → move  
• target occupied & not fixed → swap  
• target fixed → do nothing  
12. Fixed Positions 
If fixed[i] == 1: 
• cannot be modified  
• cannot be swapped  
• cannot receive new characters  
13. Multiple Sorting Processes 
Allowed per floor. 
Constraint: 
Only one process per word at a time. 
Requires per-word locking. 
14. Letter-Carrier Behaviour 
After delivery: 
• stays on destination floor  
• does NOT return  
• selects next task locally  
• may reposition if idle  

---PAGE BREAK---
15. Idle Behaviour & Repositioning 
If no task available: 
• request reposition  
• use the reposition-elevator to move to random floor 
• resume work  
16. Elevators 
Delivery Elevator 
• handles characters  
• follows direction rules  
Reposition Elevator 
• handles idle carriers  
• independent queue  
17. Termination Condition 
The program terminates ONLY when these three conditions are satisfied; 
• all words are sorted in their specified sorting floor, which means all completed 
flags are set to 1, 
• the final output file is generated, 
• summary information is printed on the terminal. 
Parent process handles shutdown.

Rules 
1. Your program must be implemented in C for a POSIX-compatible system and 
must compile and run on Debian 11 (64-bit) in VirtualBox.  
2. Your solution must use multiple processes. A solution that avoids process 
creation and simulates concurrency in a single process will not be considered a 
valid submission. 
3. All processes must be created by the parent process. Child processes must not 
create additional system-level processes. 
4. The system must initialize multiple word-carrier and letter-carrier processes per 
floor at startup. 
 
 
 
  

---PAGE BREAK---
5. Your implementation must include the following active subsystems:  
a. word-carrier processes,  
b. letter-carrier processes,  
c. sorting processes,  
d. one delivery elevator process,  
e. one reposition elevator process,  
f. one parent/coordinator process.  
6. The input file must be scanned in round-robin order by the word-carrier 
subsystem. Word selection must be synchronized so that the same word is not 
claimed by multiple processes.  
7. A word may be admitted into the system only if both:  
a. the chosen arrival floor has capacity, and  
b. the word’s sorting floor has capacity.  
8. Admission must be all-or-nothing. If both capacities cannot be reserved 
successfully, the word must be released and may be selected again later.  
9. Each character task must carry enough metadata to allow correct reconstruction, 
including at least:  
a. word_id,  
b. the character itself,  
c. original index,  
d. source floor,  
e. destination/sorting floor.  
10. Each character task must be claimed by exactly one letter-carrier process and 
delivered exactly once.  
11. Characters transported to a sorting floor must first be placed into the word’s 
sorting area at the first suitable non-fixed empty slot.  
12. Sorting processes must use the original word and the fixed/index information to 
reconstruct the word gradually.  
13. If a sorting process encounters a character already in its correct final position, it 
must leave it there and mark that index as fixed.  
14. If a character must be moved to a target position that is occupied by another non-
fixed character, only a swap operation between those two positions is allowed.  
15. Fixed positions are immutable: they can not be overwritten, they can not be 
swapped, newly arriving characters can not be placed there.  
16. A floor may have multiple sorting processes, but only one sorting process may 
operate on a specific word at a time.  
17. After delivering a character, a letter-carrier process must continue from its 
current floor. It must not automatically return to its previous floor.  
18. If a letter-carrier become idle, it should issue a reposition request and use the 
reposition elevator to move to another floor.

Notes on penalties 
• If the program crashes under normal input, the submission cannot receive a 
passing implementation score.  
• If fixed positions can be overwritten or swapped, the sorting logic is considered 
incorrect.  
• If the program terminates while work is still pending, the termination logic is 
considered incorrect. 
Submission Rules 
• Place all source files, header files, your Makefile, and your report into a directory 
named with your student number.  
• Compress that directory as a .zip archive.  
• The archive must not include compiled binaries or object files.  
• Your report must be submitted in PDF format.  
• Your report must be written in English.  
• Your Makefile must compile the program only; it must not execute it 
automatically.  
• Any submission that significantly violates the submission format may be 
rejected without full content evaluation.  
• The TAs will compile your code on their own systems.  
• Identical or suspiciously similar submissions will receive severe academic 
penalties according to course policy. 
Hints  
• Design the system architecture before writing code. This assignment is too 
complex to solve correctly by improvising.   
• Draw the process roles and the shared-memory structures on paper first.

• Clearly separate:   
o word admission,   
o character transportation,   
o sorting,   
o elevator request handling,   
o termination detection.   
• Use one clear ownership rule for every shared resource:   
o who may claim it,   
o who may modify it,   
o how it becomes available again.   
• Protect each word’s sorting state with a word-level synchronization primitive.   
• Define a consistent lock acquisition order to avoid deadlocks.   
• Never block on long waits while holding a lock that other processes need.   
• Test duplicate-character words carefully, such as words with repeated letters.   
• Test cases where:   
o preferred floors are full,   
o multiple words target the same sorting floor,   
o multiple letter-carriers deliver to the same word concurrently,   
o multiple sorters exist on the same floor, 
o no work remains and the system should terminate.  
• Check zombie cleanup explicitly after Ctrl+C.   
• Use -Wall from the beginning and keep warnings at zero.  
• Add debug logging while developing, but make sure your final output still 
conforms to the assignment requirements.
